# [실습] 스마트 홈 제어 센터 성능 최적화

현재 제공된 스마트 홈 패널 코드는 기능적으로는 잘 작동하지만, **심각한 성능 저하** 문제를 안고 있습니다.  
1초마다 업데이트되는 시계 때문에 화면 전체가 매번 다시 그려지고 있으며, 이로 인해 사용자 경험이 크게 저하되고 있습니다. 

아래 단계에 따라 문제를 진단하고, 리액트의 최적화 기법을 적용하여 부드러운 인터페이스를 완성해 보세요.

<br />

## 실습 1 요구사항

**목표: 불필요한 컴포넌트 리렌더링 현상 진단 및 방지**

**1. 성능 진단 및 문제 파악**
- 현재 `DeviceItem` 컴포넌트에는 `blockThread(50)`가  
포함되어 있어 렌더링 시 의도적인 지연(50ms)이 발생합니다.
- 상단 헤더의 **실시간 시계**가 1초마다 업데이트될 때마다,  
상태가 전혀 변하지 않은 모든 `DeviceItem`들이 매번 다시 그려지고 있습니다.
- 브라우저 개발자 도구의 **Console** 탭이나 **React DevTools → Profiler** 를  
활용하여, 시계가 움직일 때마다 어떤 컴포넌트들이 리렌더링되고 있는지 확인합니다.

**2. 컴포넌트 메모이제이션 적용**
- `DeviceItem` 컴포넌트가 부모로부터 전달받는 Props가  
변경되지 않았다면, 리렌더링을 건너뛰도록 설정합니다.
- **기대 효과**: 시계 업데이트로 인해 부모(`SmartHomePanel`)가 리렌더링되더라도,  
자식 컴포넌트들은 이전 렌더링 결과를 재사용하게 됩니다.

**3. 렌더링 로그 확인**
- `DeviceItem` 내부에 `console.log`를 추가하여 시계가 움직일 때마다  
출력되던 로그가 최적화 이후에는 더 이상 찍히지 않는지 확인합니다.

**4. UI 및 사용자 경험 체감**
- 최적화 전후의 스위치 애니메이션(Switch)이 얼마나 더 부드러워졌는지 비교해 봅니다.

## 실습 2 요구사항

**목표: 함수 참조 무결성 유지를 통한 최적화 완성**

**1. 참조 동일성 문제 발견**
- 실습 1을 완료했음에도 불구하고, **특정 기기의 스위치를 하나만 클릭해도  
모든 기기 리스트가 다시 리렌더링**되는 현상을 확인할 수 있습니다.
- **원인 분석**: 부모 컴포넌트가 리렌더링될 때마다 내부에서 선언된 `toggleDevice` 함수가  
매번 새로 생성됩니다. 이로 인해 자식에게 전달되는 `onToggle` Props의  
참조값이 바뀌어 메모이제이션이 깨지는 이유를 파악합니다.

**2. 함수 인스턴스 고정**
- `toggleDevice` 함수가 컴포넌트가 처음 마운트될 때 한 번만 생성되고,  
이후에는 동일한 참조를 유지하도록 처리합니다.
- **상태 업데이트 방식 변경**: 최신 상태를 안전하게 참조하기 위해 `setDevices` 내부에서  
**함수형 업데이트(`prev => ...`)** 패턴을 사용하고, 의존성 배열을 비워둡니다.

**3. 최적화 결과 검증**
- 이제 특정 기기를 조작했을 때, **오직 클릭한 해당 기기만** 리렌더링 로그가  
찍히고 나머지 기기들은 정지 상태를 유지하는지 최종 확인합니다.

## 실습 3 요구사항

**목표: 무거운 연산 결과의 재사용 및 접근성 강화**

**1. 파생된 상태의 계산 비용 파악**
- 현재 `connectedCount`는 `devices.filter().length`를 통해 계산됩니다.
- 비록 지금은 기기가 5개뿐이지만, 기기가 1,000개라면 시계가 1초마다 갈 때마다  
1,000번의 필터링 연산이 반복되는 낭비가 발생함을 인지합니다.

**2. 연산 결과 캐싱**
- `connectedCount` 계산 로직이 오직 `devices` 배열의 내용물이 변경될 때만 다시 실행되도록 설정합니다.
- 시계(`now`)가 바뀔 때는 이전에 계산해둔 값을 즉시 반환하여 CPU 자원을 아낍니다.

**3. 접근성(Accessibility) 및 시맨틱 마크업 보완**
- 연결된 기기 개수가 실시간으로 변할 때, 시각 장애인이 사용하는 스크린 리더가  
이 변화를 감지할 수 있도록 적절한 속성(`aria-live`)이 적용되어 있는지 확인합니다.
- `now` 시간을 표시하는 `<time>` 태그에 `datetime` 속성을 추가하여  
기계(검색 엔진, 보조 공학 기기)가 읽기 좋은 표준 형식을 제공합니다. (예: `2024-03-24T06:00`)

## 최적화 가이드

### 1. 컴포넌트 합성(Composition) - `children` 활용

가장 강력하면서도 간과되는 방법으로 "상태를 가진 부모"와 "상태와 무관한 자식"을 분리하는 방법입니다.  
부모 컴포넌트의 상태가 변하면, Props로 전달된 자식 컴포넌트도 무조건 리렌더링됩니다.

자식 컴포넌트를 `children`으로 넘기면, 부모가 리렌더링되어도 `children`으로 들어온  
컴포넌트는 참조값이 변하지 않아 리렌더링되지 않습니다.

### 2. 상태 끌어내리기

상태가 컴포넌트 전체에서 쓰이지 않는다면, 해당 상태를 사용하는  
최소 단위의 컴포넌트로 격리시키는 것입니다.

### 3. 파생된 상태

상태가 변할 때 다른 상태를 업데이트하기 위해 `useEffect` 훅을 사용하지만,  
이는 **추가적인 리렌더링(2번 렌더링)** 을 유발합니다.

이펙트 사용을 최소화하고 렌더링 과정에서 직접 계산하는 것이 성능에 유리합니다.

### 4. 상태 업데이트 배칭 및 함수형 업데이트

상태를 업데이트할 때 이전 상태에 의존적이라면? 의존성 배열에  
상태를 넣는 대신 함수형 업데이트를 사용하는 것이 좋습니다.

`useCallback`의 의존성 배열을 비울 수 있게 해주어,  
불필요한 함수 재생성을 막는 것이 핵심입니다.

### 정리

최적화의 첫 번째 단계는 `memo`를 붙이는 것이 아니라, 상태가 변할 때  
영향을 받는 컴포넌트의 범위를 최소한으로 좁히는 구조를 만드는 것입니다. 

도구(`memo`, `useMemo` 등)는 그 이후에 사용하는 보조 수단입니다.
